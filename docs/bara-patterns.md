# Bara設計パターンの踏襲

このプロジェクトは、初期設計時に参照したRustプロジェクトのリポジトリ運用・設計パターンをSSTPクライアントへ適用します。参照元の環境依存パスは記録しません。

| パターン | rust-sstpでの適用 | 理由 | 不採用理由 |
|---|---|---|---|
| Nixを正規環境にする | `flake.nix` と `.envrc` | ツール差を排除する | 開発者ごとのグローバルRustでは再現性がない |
| 小さい初期骨格 | プロトコル層、接続処理層、試験用実行基盤のみ | 依存方向を先に実証する | 未使用のオペレーティングシステム接続部や外部関数境界の抽象を先に作らない |
| ドメインとI/Oの分離 | 純粋状態機械が操作指示を返す | 高速な決定論的試験が可能 | 状態機械にソケットを埋め込まない |
| テスト駆動開発 | 状態遷移と表示の試験を同梱 | 仕様を実行可能にする | 実装後の追認試験にしない |
| ドメイン型優先 | `SstpVersion`、`SessionPhase` | 不正状態を型で制限する | 生の整数や文字列を公開しない |
| 三層の作業管理 | `TODO.md`、`docs/design-todo.md`、`docs/progress.md` | 計画・未決・実績を混ぜない | 一つの巨大メモに集約しない |
| 線形実装計画 | `TODO.md`のH番号、H内のPR単位、個別完了条件 | 小さな統合可能単位を保つ | 大見出し全体を一つのPRとして扱わない |
| 単一検証入口 | `scripts/verify` | 手元環境と継続的統合環境の意味を一致させる | 手作業の確認一覧に依存しない |
| セキュリティ基線 | cargo-deny、CVE基線、Unicode検査 | 変更差分を監査可能にする | 警告を場当たり的に無視しない |
| 構造検査 | 構成検査とドメイン型検査の処理手順 | 規約を機械的に守る | 確認者の記憶だけに頼らない |
| 定型作業手順 | `.agents/skills` | 定型作業を再現可能にする | 会話ごとに手順を再発明しない |
| エディタ設定共有 | EditorConfig/VS Code設定 | 保存時差分を減らす | 個人設定を正としない |
| 仕様・相互運用の根拠管理 | 公開仕様、既存実装、相互運用試験を記録 | 技術判断を再検証可能にする | 出所不明の知見や無記録の判断に依存しない |
| `unsafe`の隔離 | プロトコル層と接続処理層で禁止 | 監査範囲をオペレーティングシステム境界と外部関数境界へ限定 | 通信中核全域で`unsafe`を許可しない |
