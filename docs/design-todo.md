# 設計課題

未確定の設計判断だけを記録します。実装順序は `TODO.md`、完了記録は `docs/progress.md` が正です。

## D-001 非同期実行基盤の選定

- 状態: 未決
- 判断が必要になる時点: H4-1。最初のTLS・HTTP通信路を実装する前
- 比較対象: Tokio、smol系、実行基盤に依存しない待ち受け境界
- 理由: 実行基盤は取り消し、タイマー、携帯端末への組み込み、依存容量へ影響するためです。
- 不採用理由: 初期試験基盤では入出力がなく、今決めると利用側の証拠なしに公開境界を固定します。

## D-002 TLS実装基盤の選定

- 状態: 未決
- 判断が必要になる時点: H4-1。通信路の最初の縦切り前
- 比較対象: rustls、オペレーティングシステム標準TLS、実行環境別の実装基盤
- 理由: 証明書保管庫、暗号要件、Network Extension制約がオペレーティングシステムごとに異なるためです。
- 不採用理由: 単一の実装基盤を先に確定するとAppleとAndroidの資格情報統合を阻害する可能性があります。

## D-003 PPP責務の境界

- 状態: 未決
- 判断が必要になる時点: H5-1。H1でSSTPデータパケットの符号化・復号が完了した後
- 比較対象: 最小内製処理機構、成熟ライブラリの利用、オペレーティングシステムごとのPPP委譲
- 理由: 性能と移植性を左右す主要境界だからです。
- 不採用理由: pppdとPTYを共通データ経路にすると、macOSでの既知の中継費用とプロセス管理を再導入します。

## D-004 Flutter外部関数境界方式

- 状態: 未決
- 判断が必要になる時点: H7-1。H6で通信中核と全対象実行環境の接続が実証された後
- 比較対象: flutter_rust_bridge、C ABI、UniFFI相当
- 理由: Dart側の所有権、事象配信、取り消しを実例から決める必要があります。
- 不採用理由: 利用者画面より先に外部関数境界の形状を確定すると、内部状態を過剰に公開します。
