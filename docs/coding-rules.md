# コーディング規約

## 基本

- Rust 2024版を使います。
- 新しい振る舞いは失敗する試験から始めます。
- `cargo fmt`、Clippyの警告なし、全試験成功をPRの完了条件にします。
- 製品コードで `unwrap`、`expect`、`panic!`、`todo!`、`unimplemented!` を使いません。
- エラーは原因を分類し、ラップ時は `source` を保持します。

理由: 通信処理の異常系を制御フローとして扱い、再現可能な小さな変更を維持するためです。

不採用理由: 正常系だけを簡潔にしても、VPNで支配的な切断・破損・再接続時の安全性は得られません。

## ドメイン型

- 公開接続仕様で意味のある値を生の `u16`、`u32`、`String`、`Vec<u8>` として渡しません。
- 通信形式の値は検証済みの新しい型または列挙型へ変換します。
- 例外は `docs/domain-primitive-baseline.txt` に理由付きで記録します。

理由: 範囲、単位、エンディアン、秘密性を型境界で保証するためです。

不採用理由: 呼び出し側のコメント規約だけでは、不正値の生成をコンパイラが防げません。

## モジュールと責務

- `utils`、`common`、`helpers`、`types` という汎用置き場を作りません。
- クレート直下の `lib.rs` と `main.rs` は配線に限定し、実装は責務名を持つモジュールへ置きます。
- 1ファイル500行を目安とし、超過は `docs/file-size-baseline.txt` に根拠を残します。
- I/Oを行わない関数は成功時に意味のある値を返します。単なる `()` で処理結果を隠しません。

理由: 変更理由を局所化し、確認対象と所有責務を明確にするためです。

不採用理由: ファイル数だけを減らすと、状態、符号化・復号、I/Oの変更が同じ場所へ集中します。

## 危険なコード

- `sstp-protocol` と `sstp-session` は `unsafe` を全面禁止します。
- `unsafe` は将来の外部関数境界またはオペレーティングシステム接続部に限定し、各範囲へ `SAFETY:` コメントを付けます。
- 例外は `docs/unsafe-baseline.txt` で棚卸しします。

理由: メモリ安全性の監査面積をオペレーティングシステム境界だけへ閉じ込めるためです。

不採用理由: クレート全体で許可すると、性能最適化を理由にプロトコル層へ`unsafe`が拡散します。

## 性能

- 複製回数、メモリ確保、システム呼び出し、文脈切り替えを計測してから最適化します。
- PTY/TTYや外部プロセスをデータパスに置きません。
- 上限付き待ち行列と流量制御を明示し、無制限バッファーを使いません。

理由: macOSで観測されたような層間中継のボトルネックを設計段階から分離・計測するためです。

不採用理由: 非同期化や無複製という名称だけでは、オペレーティングシステムのVPN接続仕様とTLSの境界で実際に発生する複製を減らせません。
