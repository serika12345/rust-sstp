# 構成

## 依存方向

```text
利用者画面 / コマンド行インターフェース
   |
実行環境接続部 / 試験用実行基盤
   |
接続状態機械
   |
プロトコル語彙 + 通信形式の符号化・復号
```

下層から上層への依存を禁止します。`sstp-protocol` と `sstp-session` はネットワーク、ファイル、プロセス、オペレーティングシステム接続仕様へ直接アクセスしません。

理由: プロトコル判断を決定論的に試験し、I/O方式を交換可能にするためです。

不採用理由: すべてを一つの非同期反復処理に入れると、状態遷移、バッファリング、オペレーティングシステムの待ち時間を個別に測定できません。

## 初期クレート

### `sstp-protocol`

SSTPの版、メッセージ、属性、符号化・復号処理を所有します。公開接続仕様には意味を持つ型を使い、生の整数やバイト列の意味を呼び出し側へ委ねません。

理由: 通信形式の検証を一箇所へ集約し、不正状態を表現しにくくするためです。

不採用理由: `common` や `types` クレートにすると責務境界が失われ、無関係な共有物の集積所になります。

### `sstp-session`

接続、SSTP制御、切断の状態遷移と、副作用として必要なアクションを値として返します。

理由: 状態遷移と実際のソケットI/Oを分け、試験を高速かつ再現可能にするためです。

不採用理由: 状態機械自身にソケットを持たせると、時間切れや再送の試験が実時間とオペレーティングシステムへ依存します。

### `sstp-harness`

コマンド行引数、表示、将来の試験用I/Oを所有します。製品の利用者画面ではありません。

理由: 最小の実行可能な縦切りで依存方向と状態機械の振る舞いを検証するためです。

不採用理由: 最初からFlutterを結合すると、外部関数境界と利用者画面の問題がプロトコル検証へ混ざります。

## 将来の境界

TLSとHTTPによる通信路、PPP処理機構、Linux・Android・Apple接続部、Flutter外部関数境界は、利用側の具体的要求が生じた時点で別クレートとして追加します。

理由: 実際の差分から抽象を抽出し、不要な仮想化を避けるためです。

不採用理由: 先回りした汎用`trait`群は、所有権、流量制御、取り消し方式を誤って固定する可能性があります。
