# rust-sstp 設計計画

作成日: 2026-07-20  
文書状態: 設計基準 / H0完了後

## 0. この文書の読み方

本書の各設計項目は、次の3点を必ず記載する。

- **決定**: 採用する設計。
- **理由**: その設計が目的に適している理由。
- **不採用理由**: 主な代替案を採用しない理由。代替案自体を否定するのではなく、本プロジェクトの目的と現段階に適さない理由を示す。

本書の目的は、単に動作するSSTPクライアントではなく、以下を同時に満たすクライアントを設計することである。

- 高速: `pppd`、PTY、過大なキュー、パケット単位外部関数境界など、既知の人工的な処理停滞箇所を持たない。
- 安全: Rustの型・所有権を活用し、入力検証、秘密情報管理、TLS検証、SSTP Crypto Bindingを省略しない。
- クロスプラットフォーム: Linux、Android、macOS、iOSで同一のプロトコルコアを使用する。
- オペレーティングシステム統合: 各オペレーティングシステムが提供する正式なVPNインターフェースとライフサイクルに従う。
- 保守可能: 状態機械、I/O、オペレーティングシステム固有処理、利用者画面を分離し、単体試験と相互運用試験を容易にする。

## 1. 製品スコープ

### 1.1 初期対象はIPv4、MS-CHAPv2、PAPとする

**決定**  
最初の実用最小実用製品はIPv4、LCP、IPCP、PAP、MS-CHAPv2、SHA-256 Crypto Bindingを実装する。PAPは開発・相互運用確認用とし、製品上の推奨既定はMS-CHAPv2とする。

**理由**
SoftEtherおよび多くのSSTP環境との最小限の相互運用範囲を確保しつつ、SSTP固有の重要部分であるMS-CHAPv2鍵導出とCrypto Bindingを初期段階で検証できる。

**不採用理由: 初版からEAP-TLS、PEAP、TTLS、TEAPをすべて実装する**  
EAP方式ごとに内側TLS、証明書選択、鍵素材の取り出し、断片化、方式固有の状態機械が必要になる。基礎となるSSTP/PPP状態機械が安定する前に追加すると、障害の切り分けが困難になる。

### 1.2 PPP暗号化ではなくTLSを外側の機密性保護とする

**決定**  
SSTPの仕様どおりTLSをトンネルの機密性・完全性保護に使用する。MPPEデータ暗号化を初期データパスへ追加しない。ただしRFC 3079の鍵導出はCrypto Bindingのため実装する。

**理由**  
SSTPはTLS内でPPPを運ぶ。MPPE鍵素材はCrypto Bindingに必要だが、同じデータをさらに暗号化することは通常不要で、中央処理装置負荷と実装面積を増やす。

**不採用理由: PPPデータもMPPEで二重暗号化する**  
相互運用上必須でない限り、二重暗号化は速度を落とし、古い暗号構成への依存を増やす。対象サーバで必須と判明した場合に、互換機能として別途検討する。

## 2. 全体構成

### 2.1 制御面とデータ面を分離する

**決定**  
Flutter利用者画面を制御面、Rust SSTP処理機構とオペレーティングシステムのVPN接続部をデータ面とする。Flutterには接続操作、設定、状態、集約統計、診断事象だけを公開する。

```text
Flutter利用者画面
  │  接続指示・状態・統計
  ▼
実行環境制御処理
  │
  ▼
Rust SSTP処理機構 ── Rustプロトコル中核
  │
  ▼
オペレーティングシステムのVPNパケット接続口
```

**理由**  
Dart分離実行単位、Flutter実行環境チャネル、利用者画面の生存期間管理をパケット経路から除外できる。iOSとmacOSでは利用者画面アプリとPacket Tunnel拡張機能が別プロセスであり、オペレーティングシステムが管理するトンネルを利用者画面停止後も継続できる。

**不採用理由: IPパケットをFlutter/Dartへ渡す**  
パケットごとに外部関数境界、オブジェクト生成、複製、分離実行単位の実行順制御が発生する。利用者画面の自動メモリ回収や描画負荷がトンネル性能と安定性に影響し、Appleの拡張機能構成にも適合しない。

### 2.2 オペレーティングシステム非依存中核と薄い実行環境接続部に分ける

**決定**  
プロトコル状態機械をオペレーティングシステム接続仕様から切り離す。オペレーティングシステム差分はパケットI/O、外側TCP接続、IP・DNS・経路設定、資格情報取得、生存期間通知に限定する。

**理由**  
SSTP、PPP、認証、Crypto Bindingを全オペレーティングシステムで一度だけ実装できる。オペレーティングシステム 接続仕様変更や署名方式変更がプロトコルロジックへ波及しない。

**不採用理由: 各オペレーティングシステムで独立したSSTPクライアントを実装する**  
相互運用バグとセキュリティ修正を5回実装することになる。実装差が増え、あるオペレーティングシステムだけCrypto BindingやEAP挙動が異なる状態になりやすい。

### 2.3 最初からマイクロクレート化しすぎない

**決定**  
初期作業空間の主要クレートは`sstp-core`、`sstp-engine`、`sstp-ffi`、`control-protocol`の4つに留める。

```text
crates/
  sstp-core/          # 符号化・復号処理、状態機械、認証、Crypto Binding
  sstp-engine/        # TLS、非同期I/O、バッファ、流量制御
  sstp-ffi/           # AppleとAndroid向けの狭いC応用バイナリインターフェース
  control-protocol/   # 利用者画面、サービス、拡張機能間の制御メッセージ
platforms/
  linux/
  android/
  apple/
flutter/
```

**理由**  
責務境界を明確にしつつ、初期開発でクレート間接続仕様変更の負担を増やさない。独立した再利用・公開・構築条件が生じたものだけ後から分割できる。

**不採用理由: SSTP、PPP、LCP、IPCP、認証方式ごとにクレートを分ける**  
初期段階では状態・エラー・バッファ型が頻繁に変わる。細分化は公開接続仕様の固定を早め、変更時の作業量と依存関係を増やす。

## 3. プロトコルコア

### 3.1 同期的な事象・操作指示反応器とする

**決定**  
`sstp-core`はイベントを受け、実行すべきアクションを返す同期接続仕様とする。非同期実行基盤、ソケット、TUN、待機を持たせない。

```rust
pub enum Event<'a> {
    SstpBytes(&'a [u8]),
    IpPackets(&'a [PacketView<'a>]),
    TimerExpired(TimerId),
    NetworkConfigured,
    Command(Command),
}

pub enum Action {
    SendSstp(PacketBatch),
    SendIp(PacketBatch),
    ApplyNetworkConfig(NetworkConfig),
    SetTimer(TimerId, Duration),
    CancelTimer(TimerId),
    EmitStatus(StatusEvent),
    Disconnect(DisconnectReason),
}
```

**理由**  
同じ事象列から同じ結果を得られ、時間切れや異常入力を実時間なしで再現できる。ファズ試験、性質試験、pcap再生が容易になる。TokioやAppleコールバック方式の違いも中核へ持ち込まない。

**不採用理由: コアの各状態を`async fn`として直接実装する**  
I/O待ちと状態遷移が混ざり、テストで実実行基盤とタイマー操作が必要になる。キャンセル時にどの非同期結果が状態を所有するか不明瞭になりやすい。

### 3.2 セッション状態の所有者を1つにする

**決定**  
`SessionCore`だけがSSTP、PPP、認証、IPCP/IPv6CP状態を変更できる。状態変数を複数処理単位と`Arc<Mutex<_>>`で共有しない。

**理由**  
状態遷移の順序が明確になり、Call Abort、PPP Terminate、時間切れ、ソケット切断が競合しない。ロック順序や部分更新による不整合を排除できる。

**不採用理由: SSTP、LCP、認証、IPCPを独立処理単位として動かす**  
各プロトコルはフェーズと鍵素材を相互に受け渡す。独立処理単位にすると、キャンセル、再送、認証成功直後のCrypto Bindingなどで複雑な同期が必要になる。

### 3.3 状態を明示的なenumで表現する

**決定**  
SSTPとPPPの状態はenumで表し、不正な組み合わせを可能な限り型で表現不能にする。

```rust
enum SstpState {
    Disconnected,
    HttpNegotiating,
    WaitCallConnectAck,
    PppNegotiating,
    WaitCryptoBinding,
    Running,
    Disconnecting,
}

enum PppPhase {
    Dead,
    Establish,
    Authenticate,
    Network,
    Running,
    Terminate,
}
```

**理由**  
任意の真偽値群よりレビューしやすく、網羅的matchにより新状態追加時の処理漏れをコンパイラが検出できる。

**不採用理由: `is_connected`、`is_authenticated`等の真偽値を並べる**  
`connected=false`なのに`authenticated=true`など、意味のない組み合わせが発生する。異常系の再現と修正が難しくなる。

### 3.4 通信形式の符号化・復号処理と状態判断を分離する

**決定**  
符号化・復号処理はバイト列の解析/encodeだけを担当し、パケットを受理すべき状態かどうかは状態機械が判断する。

**理由**  
符号化・復号処理をファズ試験しやすく、同じ符号化・復号処理をテストツールやpcap解析でも使える。状態依存エラーと形式エラーを区別できる。

**不採用理由: 解析関数内で状態遷移まで実行する**  
入力検証と副作用が結合し、途中まで解析した後に状態だけ変わる問題が起きやすい。単体試験も複雑になる。

### 3.5 逐次処理 復号器を採用する

**決定**  
SSTP入力は連続byte ストリームとして扱い、1回のreadに0.5個、1個、複数個のSSTP パケットが含まれる全ケースに対応する。

**理由**  
TCP区間、TLS 記録、SSTP パケットの境界は一致しない。断片化・結合を正しく扱うことが相互運用性と安全性の基礎になる。

**不採用理由: 1回のソケット readを1個のSSTP パケットとみなす**  
LANや特定TLS実装では偶然動いても、実ネットワーク、異なる記録 容量、プロキシ経由で破綻する。

### 3.6 長さ上限を全階層で持つ

**決定**  
SSTP パケット、属性、PPP パケット、LCP 選択肢、EAP断片、文字列に明示的な最大長を設け、算術は検査付き演算を使う。

**理由**  
外部入力によるメモリ過剰確保、整数あふれ、無限待機を防ぐ。Network Extensionのようなメモリ制限環境でも予測可能になる。

**不採用理由: `Vec`を入力長に合わせて無条件に拡張する**  
攻撃者または壊れたサーバが巨大な長さを宣言した場合にメモリ枯渇やサービス停止を起こせる。

## 4. 非同期I/Oエンジン

### 4.1 コアと実行基盤の間に`sstp-engine`を置く

**決定**  
`sstp-engine`がTCP/TLS read、パケットI/O、タイマー、操作指示実行、キャンセル、再接続を担当する。

**理由**  
プロトコル状態を純粋に保ちながら、オペレーティングシステムごとのI/O方式を統一した実行モデルへ変換できる。性能計測点も処理機構に集約できる。

**不採用理由: 各実行環境 接続部が直接`sstp-core`を駆動する**  
アクション優先度、キュー、TLS 送信確定、タイマー、エラー処理がオペレーティングシステムごとに重複し、性能と挙動が分岐する。

### 4.2 I/Oポートを最小限にする

**決定**  
主要な抽象化を`PacketIo`、`OuterConnector`、`NetworkConfigurator`の3つに限定する。資格情報と時計・乱数は小さな補助インターフェースとして扱う。

```rust
pub trait PacketIo {
    async fn receive(&mut self, 一括処理: &mut PacketBatch) -> io::Result<usize>;
    async fn send(&mut self, 一括処理: &PacketBatch) -> io::Result<()>;
}

pub trait OuterConnector {
    type Stream: AsyncRead + AsyncWrite + Unpin;
    async fn connect(&mut self, endpoint: &Endpoint) -> io::Result<Self::Stream>;
}

pub trait NetworkConfigurator {
    async fn apply(&mut self, 設定: &NetworkConfig) -> Result<(), PlatformError>;
    async fn clear(&mut self) -> Result<(), PlatformError>;
}
```

**理由**  
各オペレーティングシステム差分が実際に存在する境界だけを抽象化する。インターフェース数を抑え、模擬実装も小さくする。

**不採用理由: ファイル、ソケット、DNS、経路、タイマー、ログ処理ごとに多数の`trait`を作る**  
抽象化のための抽象化になり、実装追跡が難しくなる。実際に差し替えが必要な依存だけをインターフェース化する。

### 4.3 TLS書き込み処理は1つだけにする

**決定**  
SSTP制御、PPP交渉、IPデータの全出力を単一の書き込み処理へ集約し、優先度付きの有界待ち行列で処理する。

**理由**  
TLSストリームへの書き込み順を保証し、複数処理単位の相互排他競合を避ける。混雑時にも切断、応答確認、認証応答をデータより優先できる。

**不採用理由: 上り処理、応答確認処理、PPP処理が同じTLS書き込み処理を相互排他で共有する**  
ロック待ちと優先度逆転が起きる。切断メッセージが大量データの後ろに並び、終了処理が遅れる可能性がある。

### 4.4 固定待機と短周期ポーリングを使わない

**決定**  
ソケット、TUN、ファイル記述子、Apple コールバックの準備完了通知とタイマーを使う。通常ループに固定`1ms 時間切れ`、`10us 待機`、無条件`yield_now`を置かない。

**理由**  
待機時の中央処理装置使用量、携帯端末の起床回数、負荷時の不必要な遅延を減らす。事象があるときだけ処理する。

**不採用理由: 短い待機で単純な待ち受け反復処理を作る**  
実装は一見簡単だが、低遅延を狙うほど中央処理装置を消費し、使用量を抑えるほど遅延が増える。オペレーティングシステムの準備完了機構の方が目的に適する。

## 5. パケットバッファとコピー削減

### 5.1 先頭余白付き再利用バッファーを使う

**決定**  
IPパケットを格納する際に先頭へ16〜32バイト程度の余白を確保し、PPPとSSTPのヘッダーを前方へ追加できる`PacketBuf`を使用する。

```text
受信直後:
| 先頭余白            | IPパケット                       |

カプセル化後:
| SSTPヘッダー | PPPプロトコル | IPパケット              |
```

**理由**  
IPパケットをPPP用`Vec`、さらにSSTP用`Vec`へ複製する処理を避けられる。MTU程度のバッファーを再利用できる。

**不採用理由: 各層で新しい`Vec<u8>`を生成する**  
パケットレートに比例してメモリ確保と全体コピーが増える。小パケットが多い通信で中央処理装置負荷が顕著になる。

### 5.2 完全無複製を必須目標にしない

**決定**  
目標を「オペレーティングシステム境界で最大1回の複製、Rust内部で可能な限り再利用」とする。完全無複製は必須条件にしない。

**理由**  
rustlsの暗号化バッファー、Appleの`Data`、Android TUN接続仕様など、所有権と寿命が異なる境界がある。安全性と単純さを保った1回の複製は、多くの場合十分高速である。

**不採用理由: unsafeとオペレーティングシステム内部バッファー参照を多用して完全無複製にする**  
バッファー 生存期間、非同期コールバック、キャンセル時解放が複雑になり、解放後使用の危険が増える。計測前の最適化として費用対効果が悪い。

### 5.3 パケットをバッチ単位で扱う

**決定**  
`PacketIo`、外部関数境界、オペレーティングシステム書き込みは複数パケットの`PacketBatch`を基本単位とする。

**理由**  
システム呼び出し、外部関数境界、コールバック、処理単位の起床回数を減らす。Appleの`NEPacketTunnelFlow`も複数パケット接続仕様を提供する。

**不採用理由: 常に1パケットずつ処理する**  
低帯域では動作するが、毎秒パケット数が多い環境では固定処理費用が支配する。携帯端末でも起床回数の増加が電池消費につながる。

### 5.4 キューをバイト数で有界化する

**決定**  
制御待ち行列は小さく独立させ、データ待ち行列は初期値256KiB〜1MiB程度の総バイト数上限を持たせる。値は性能測定で調整する。

**理由**  
1500バイトのパケットと小パケットでは同じ要素数でもメモリ量が異なる。バイト上限ならメモリ予算と待ち時間を予測しやすい。

**不採用理由: 30,000要素など大きなチャネルを用意する**  
輻輳を隠して遅延とメモリ使用量を増やす。古いパケットが長時間残り、TCPの流量制御とも整合しない。

## 6. TLSと外側TCP

### 6.1 TLS実装はrustlsを第一候補とする

**決定**  
TLS 1.2とTLS 1.3にはrustlsを使用し、オペレーティングシステムの信頼証明書保管庫との統合には`rustls-platform-verifier`を評価・採用する。

**理由**  
Rust中心の実装でメモリ安全性を維持でき、クロスプラットフォームに同じTLS動作を持てる。オペレーティングシステムの企業CAや証明書制約との統合も可能である。

**不採用理由: OpenSSLを全オペレーティングシステムで直接リンクする**  
Android/iOS/macOSでのビルド、バイナリ配布、ABI、アップデートが複雑になる。既存C実装との互換性には有利だが、本計画の安全・移植性目標にはrustlsの方が適する。

### 6.2 証明書検証を既定で必須にする

**決定**  
通常のPKIとホスト名検証を既定とする。自己署名環境には明示CA追加または証明書/SPKI 固定を提供する。検証完全無効化は製品版の通常設定に置かない。

**理由**  
SSTPの外側TLSサーバ認証はCrypto Bindingと組み合わせて中間者攻撃を防ぐ基礎である。ユーザー名・パスワードを扱うため、便利さより安全な既定値を優先する。

**不採用理由: `insecure=true`で全証明書を受理する**  
利用者が設定を戻さず恒久運用する可能性が高く、SSTPのセキュリティモデルを破壊する。診断用途でも明示CAや固定で代替できる。

### 6.3 パケットごとにTLS 送信確定しない

**決定**  
制御パケット、サイズ閾値、短い遅延期限、キュー空のいずれかで送信確定する。データパケット単位では送信確定しない。

**理由**  
複数SSTP パケットをTLS 記録へ集約し、システム呼び出しと暗号処理の固定費を削減する。一方で短い期限により対話通信の遅延を抑える。

**不採用理由: 各IPパケットの直後に送信確定する**  
小さいTLS 記録とwrite システム呼び出しが大量発生する。過去の実装で送信確定位置や固定遅延が性能へ大きく影響したこととも整合しない。

### 6.4 TCP_NODELAYを有効化し、集約はアプリ側で制御する

**決定**  
外側SSTP TCP ソケットは原則`TCP_NODELAY`を有効にし、必要な集約は処理機構の短時間一括化で制御する。

**理由**  
Nagleと遅延ACKの相互作用による予測しにくい遅延を避ける。制御パケットを即時送信しつつ、データは明示的にまとめられる。

**不採用理由: Nagleへ全面的に集約を任せる**  
オペレーティングシステムとネットワーク条件により待機時間が変わり、制御パケットとデータパケットを区別できない。計測可能なアプリ側方針の方が調整しやすい。

### 6.5 TCP-over-TCPを実装上の制約として明示する

**決定**  
損失回線での性能低下をSSTPプロトコル固有の制約として文書化し、ベンチマークへパケット 損失率と往復時間を含める。

**理由**  
SSTPの外側TCPと内側TCPの輻輳・再送が干渉する問題は、Rust化や無複製では解消できない。実装上の遅さとプロトコル上限を区別する必要がある。

**不採用理由: LAN上の最大スループットだけで高速性を評価する**  
LANではメモリ確保や暗号速度しか見えず、実利用で問題になる損失率、揺らぎ、先頭待ちによる後続処理の停止を評価できない。

## 7. PPPとSSTPセキュリティ

### 7.1 `pppd`、PTY、TTYへ依存しない

**決定**  
LCP、認証、IPCP、IPv6CPをRustコア内で実装し、外部`pppd`プロセスを使用しない。

**理由**  
macOSで観測されたTTY速度由来の処理停滞箇所、プロセス間複製、文脈切り替え、オペレーティングシステムごとの`pppd`差異を排除できる。iOS/Androidでも同一PPP実装を使える。

**不採用理由: 成熟した`pppd`を再利用する**  
Linuxでは十分高速でも、Apple 携帯端末環境では利用できず、macOSでは実装依存問題がある。クロスプラットフォームの中核としては成立しない。

### 7.2 SSTP Crypto Bindingを最小実用製品必須とする

**決定**  
MS-SSTP仕様に従い、サーバ証明書ハッシュ、使い捨て値、HLAK、CMK、Compound MACを実装する。SHA-256を優先する。

**理由**  
TLSサーバ認証とPPPクライアント認証を暗号学的に結合するSSTPの必須機能である。省略すると中間者によるPPP relayを防げない。

**不採用理由: 接続できるサーバ向けに空のCall Connectedを送る**  
寛容なサーバでは接続しても仕様違反かつ安全性を損なう。後付けすると認証接続仕様と鍵寿命を大きく変更するため、初期設計に含めるべきである。

### 7.3 認証方式を共通インターフェースに閉じ込める

**決定**  
PAP、MS-CHAPv2、EAP方式は`AuthMethod`として、入力パケット、出力パケット、完了結果、Crypto Binding用鍵素材を統一的に返す。

```rust
pub struct AuthSuccess {
    pub peer_identity: PeerIdentity,
    pub binding_key: Option<SecretBytes>,
}
```

**理由**  
SSTP層は認証方式の内部アルゴリズムを知らず、HLAK生成に必要な鍵素材だけを受け取れる。EAP追加時にSSTP状態機械を変更しすぎない。

**不採用理由: MS-CHAPv2処理をSSTP状態機械へ直接書く**  
PPP認証とSSTP制御が密結合になり、EAP-TLS等を追加するたびに中心状態機械が肥大化する。

### 7.4 秘密情報を専用型で扱う

**決定**  
パスワード、NT ハッシュ、MasterSendKey、MasterReceiveKey、HLAK、CMKを`SecretBytes`等の専用型で表現し、Debug表示を禁止し、不要後にzeroizeする。

**理由**  
誤ったログ出力、複製、長期保持を防ぐ。認証完了後に必要な鍵だけを明確に残せる。

**不採用理由: 通常の`String`や`Vec<u8>`を使う**  
派生`Debug`、エラーメッセージ、複製による複製が容易で、消去時期も不明確になる。

## 8. 実行環境統合

### 8.1 AppleはPacket Tunnel拡張機能を正式経路とする

**決定**  
iOSとmacOSはSwiftの`NEPacketTunnelProvider`が生存期間、`NEPacketTunnelFlow`、IP・DNS・経路・MTU設定を担当し、Rust静的ライブラリを同じ拡張機能内で実行する。

**理由**  
AppleがカスタムVPN向けに提供する正式接続仕様であり、オペレーティングシステム設定画面、VPN状態、オンデマンド接続、利用者画面アプリ停止後の継続動作へ統合できる。

**不採用理由: macOSの`utun`を直接開き、iOSも非公開ファイル記述子取得へ寄せる**  
非公開または非推奨の挙動へ依存し、App Store審査と将来のオペレーティングシステムで破綻する。iOSには任意TUN作成の公開接続仕様がない。

### 8.2 Appleのパケット境界は一括処理型C応用バイナリインターフェースにする

**決定**  
SwiftからRustへ複数パケットを一度に渡し、RustからSwiftへも一括コールバックで返す。不透明な接続識別子を持つ狭いC応用バイナリインターフェースを定義する。

```c
sstp_session_create(...)
sstp_session_start(...)
sstp_session_submit_packet_batch(...)
sstp_session_notify_path_change(...)
sstp_session_stop(...)
sstp_session_destroy(...)
```

**理由**  
応用バイナリインターフェースの面積を小さくし、SwiftとRust間の呼び出し回数を減らす。Packet Tunnel拡張機能側の所有権と取り消しを明示できる。

**不採用理由: Rustの複雑な型や非同期結果をSwiftへ直接公開する**  
応用バイナリインターフェースの安定性、所有権、実行器の対応が複雑になる。外部関数境界は単純なバッファー、接続識別子、コールバックに限定する方が監査しやすい。

### 8.3 AndroidはVpnServiceのTUNファイル記述子をRustへ渡す

**決定**  
Kotlin `VpnService`が利用者同意、前景サービス、常時接続、ネットワーク変更、TUN確立を担当し、`detachFd()`したTUNファイル記述子をRust処理機構へ渡す。

**理由**  
Android公式接続仕様がネイティブ空間でのファイル記述子利用を認めており、Rustが直接読み取りと書き込みを行える。パケットごとのJavaネイティブインターフェース往復を避けられる。

**不採用理由: KotlinでTUNを読み、毎パケットJavaネイティブインターフェースでRustへ渡す**  
メモリ確保、Javaネイティブインターフェース、コルーチン実行順制御がデータ経路に追加され、毎秒パケット数が多い場合の処理停滞箇所になる。

### 8.4 Androidの外側ソケットは必ず`protect()`する

**決定**  
未接続ソケットのファイル記述子をRustで作成してKotlinへ渡し、`VpnService.protect(fd)`成功後にRustが接続する方式を第一候補とする。

**理由**  
SSTP TCP接続自身がVPN経路へ再流入する無限反復を防ぎつつ、TCPとTLSの本体をRustへ統一できる。

**不採用理由: `protect`前にRustが接続する**  
全通信トンネル経路の適用後には接続先SSTPサーバーへのパケットがTUNへ戻り、接続不能または反復状態になる可能性がある。

### 8.5 LinuxはTUNとnetlinkを使うRust常駐処理から開始する

**決定**  
最初の実行環境をLinuxとし、Linux TUN文字デバイス、netlink、Unixドメインソケットを使う画面なし常駐処理を作る。

**理由**  
開発、pcap形式の記録、`tc netem`、SoftEther試験、ファズ試験の再現が容易で、オペレーティングシステム署名や携帯端末拡張機能の制限へ対応する前にプロトコル中核を安定させられる。

**不採用理由: 最初からiOSで開発する**  
拡張機能メモリ、署名、権限設定、別プロセス、実機診断がプロトコル不具合の切り分けを難しくする。

### 8.6 NetworkManager統合は後付け接続部とする

**決定**  
Linuxの最小実用製品は、独立常駐処理とコマンド行インターフェースで完成させ、NetworkManager統合は安定後に追加する。

**理由**  
プロトコル検証をD-Bus、追加機能の応用バイナリインターフェース、配布物差から切り離せる。

**不採用理由: 初期コアをNetworkManager 追加機能として実装する**  
試験と配布が特定デスクトップ環境に依存し、画面なし継続的統合環境や他オペレーティングシステムへの移植に不要な制約が入る。

## 9. Flutterと制御プロトコル

### 9.1 Flutterには統一実行環境追加機能を提供する

**決定**  
Dart側に接続、切断、接続設定管理、状態通知流、統計通知流、診断情報を提供する実行環境追加機能を作る。

**理由**  
Flutterコードからはオペレーティングシステムごとのサービスと拡張機能の差を隠せる一方、各オペレーティングシステムの正式なプロセス方式を維持できる。

**不採用理由: 全オペレーティングシステムでFlutter Rust Bridgeから同じRust オブジェクトを直接呼ぶ**  
Apple Packet Tunnel拡張機能はFlutterと別プロセスであり、同一メモリ上のオブジェクトを共有できない。AndroidもVpnServiceの生存期間が利用者画面のActivityと異なる。flutter_rust_bridgeだけでは生存期間を統一できない。

### 9.2 制御メッセージは版管理し、低頻度にする

**決定**  
制御プロトコルは版を持つ小さな要求・応答・事象の構造定義とし、JSONまたはCBOR等の単純な形式を使用する。パケットデータは含めない。

**理由**  
異なる版の利用者画面と実装基盤の間で互換性判断ができ、診断情報も読みやすい。制御面は低頻度なので直列化性能は問題にならない。

**不採用理由: 独自バイナリ配置を最初から最適化する**  
制御面の性能利益が小さい一方、互換性、デバッグ、言語間実装の負担が増える。

### 9.3 統計を集約して通知する

**決定**  
バイト数とパケット数、待ち行列の深さ、往復時間、再接続回数、現在段階を処理機構で集約し、利用者画面へ毎秒1〜4回程度通知する。

**理由**  
性能分析に必要な情報を提供しつつ、利用者画面更新とプロセス間通信をデータ量から切り離せる。

**不採用理由: パケットごとにログまたは進捗事象を送る**  
利用者画面とプロセス間通信がデータ経路の負荷に比例し、ログ自体が処理停滞箇所になる。秘密情報や通信内容を漏らす危険も高まる。

## 10. エラー、キャンセル、再接続

### 10.1 エラーを段階付きで分類する

**決定**  
DNS、TCP、プロキシ、TLS、HTTPとSSTP、PPP、認証、Crypto Binding、ネットワーク設定、パケットI/Oを別のエラー分類にする。

**理由**  
利用者向け表示と診断を分けられ、相互運用失敗の層を特定できる。再試行可能性も分類できる。

**不採用理由: すべて文字列エラーとして伝える**  
利用者画面の翻訳、再試行判断、遠隔測定集計が文字列解析に依存し、将来変更に弱くなる。

### 10.2 取り消しをセッション全体で一方向に伝播させる

**決定**  
停止要求、オペレーティングシステムの権限取り消し、ソケット終端、致命的プロトコルエラーのいずれかで接続処理の取り消しを発火し、読み取り処理、書き込み処理、パケットI/O、タイマーを順序立てて終了する。

**理由**  
処理単位が残存して古いTUNやソケットへ書くことを防ぐ。終了理由を1つに確定できる。

**不採用理由: 各処理単位が独自に終了・再接続する**  
旧読み取り処理と新接続処理が同時に動く、二重経路設定、複数書き込み処理などの競合が発生しやすい。

### 10.3 再接続方針は処理機構外の方針として持つ

**決定**  
処理機構は1回の接続処理を確立・終了し、指数的な再試行間隔、常時接続、利用者停止、ネットワーク変更時再接続は実行環境監督処理が決定する。

**理由**  
オペレーティングシステムごとのバックグラウンド制約と利用者設定が異なる。プロトコル処理機構を再接続方針から分離できる。

**不採用理由: `sstp-core`が永久に自動再接続する**  
利用者停止、資格情報失敗、オペレーティングシステムの権限取り消しでも再試行し続ける危険がある。中核がオペレーティングシステムの生存期間を知る必要も生じる。

## 11. 安全性設計

### 11.1 中核では`unsafe`を禁止する

**決定**  
`sstp-core`は`#![forbid(unsafe_code)]`を目標とし、`unsafe`はファイル記述子、AppleのC応用バイナリインターフェース等を扱う実行環境モジュールと外部関数境界モジュールへ隔離する。

**理由**  
外部入力を解析する最重要部分を安全なRustに限定できる。`unsafe`箇所を小さく監査できる。

**不採用理由: 性能のため中核にも`unsafe`解析器を導入する**  
安全なスライス解析で十分な性能が期待できる。性能測定前にメモリ安全性を犠牲にする根拠がない。

### 11.2 資格情報の保存をオペレーティングシステムへ委譲する

**決定**  
Apple Keychain、Android Keystore対応保存機構、Linux Secret Service等を実行環境制御処理が使用し、中核には接続時だけ秘密を渡す。

**理由**  
オペレーティングシステムのアクセス制御、生体認証、端末暗号化、管理対象設定へ統合できる。

**不採用理由: Flutterの設定ファイルや共通SQLiteへパスワードを保存する**  
平文・弱い暗号化・バックアップ流出の危険があり、オペレーティングシステムごとの安全機能を利用できない。

### 11.3 解析器のファズ試験を初期実装の完了条件にする

**決定**  
SSTP、属性、PPP、LCP、IPCP、CHAPとEAPの符号化・復号処理にlibFuzzerと`cargo-fuzz`の試験対象を作る。

**理由**  
VPNクライアントは認証前から外部サーバーの任意入力を処理する。長さ、断片化、未知選択肢、重複項目を自動探索する価値が高い。

**不採用理由: 単体試験だけで解析器を保証する**  
手書き事例では組み合わせ爆発と予想外の長さ符号化を網羅できない。

## 12. テスト戦略

### 12.1 状態機械を決定論的に単体試験する

**決定**  
模擬時計、固定乱数、入力事象列を使い、期待操作指示と最終状態を検証する。

**理由**  
再送、時間切れ、識別子の周回、重複パケット、切断競合を高速かつ再現可能に試験できる。

**不採用理由: 実サーバとの端点間試験試験だけに依存する**  
障害時にクライアント、サーバ、ネットワークのどこが原因か分からず、まれな 状態を意図的に作れない。

### 12.2 Windows RRASとSoftEtherを基準サーバにする

**決定**  
Windows Server RRASを事実上の互換基準、SoftEtherを自動化しやすい公開試験サーバとして使用する。後にMikroTik、Azure P2Sを追加する。

**理由**  
Microsoft実装との互換性と、継続的統合環境可能なオープン実装の両方を確認できる。

**不採用理由: SoftEtherだけで準拠性を判断する**  
SoftEther固有の寛容さや挙動へ適合しただけの可能性があり、Windows RRASで失敗するリスクが残る。

### 12.3 既存クライアントを性能基準にする

**決定**  
Linux`sstp-client`、Windows標準、macOS/iOS SSTP Connect、Android Open SSTP Clientを比較対象にする。

**理由**  
新実装が理論上速いという推測ではなく、同じ端末・サーバ・認証・MTU・回線条件で実測できる。

**不採用理由: Rust内部の小規模性能測定だけで高速性を主張する**  
符号化・復号処理速度が高くても、TLS write、待ち行列、オペレーティングシステムのパケットI/O、経路設定、TCP-over-TCPが支配すれば製品性能は向上しない。

### 12.4 損失と遅延を含む性能試験を行う

**決定**  
往復時間 1/20/100/250ms、損失率 0/0.1/1/3%、揺らぎ、帯域非対称、Wi-Fi/セルラー切替を試験する。

**理由**  
SSTPのTCP-over-TCP特性、流量制御、再接続、待ち行列遅延を評価できる。

**不採用理由: 損失率 0%のLANだけで測る**  
実装の最大暗号速度しか見えず、SSTPが最も苦手とする条件を見落とす。

## 13. 依存関係とライセンス

### 13.1 プロジェクトライセンスはApache-2.0とする

**決定**  
新規コードは`Apache-2.0`で提供する。ライセンス本文はリポジトリ直下の`LICENSE`を正とする。

**理由**  
商用・オープンソースソフトウェアの双方で利用しやすい寛容なライセンスであり、明示的な特許許諾と特許訴訟時の終了条項を持つため、VPNプロトコル・暗号・オペレーティングシステム統合を扱う本プロジェクトの条件をMITより明確にできる。

**不採用理由: MITとの二重ライセンスまたはGPLとする**  
MITとの二重ライセンスでは利用者が特許条項を持たないMITだけを選べるため、Apache-2.0へ統一する意図が弱まる。GPLは製品・開発用部品利用の選択肢を狭め、GPLコードをApache-2.0の成果物へ取り込めないため採用しない。

### 13.2 依存関係を少数・明示的に保つ

**決定**  
TLS、暗号基本機能、zeroize、bytes、実行基盤、オペレーティングシステム接続処理等の必要依存だけを採用し、`cargo-deny`、`cargo-audit`、ソフトウェア部品表を継続的統合環境へ入れる。

**理由**  
VPNクライアントは高権限かつ長期稼働する。supply-chain面積とアップデート負担を把握する必要がある。

**不採用理由: 高機能な汎用ネットワーク 枠組みを大量に組み合わせる**  
不要機能、unsafe、実行環境非対応、重複TLS 層を持ち込み、携帯端末向け構築と監査を難しくする。

この節は設計時に整理した機能群と依存関係を示す参考資料であり、実装順序やPR単位の正ではない。現在の線形な実装順序、個別の完了条件、含む範囲、除外範囲、検証、停止条件は `TODO.md` のH0からH8を唯一の正とする。

**理由:** 設計上の長期像と、現在の実装証拠から細分化したPR単位を分離し、順序変更時に複数の計画を同期する必要をなくすため。

**不採用理由: この節と`TODO.md`の両方を実装順序として保守する**
粒度と更新時期が異なる二つの順序表は、次に着手すべき作業と完了判定を矛盾させるため。

### 段階 0: 仕様固定と試験素材

**決定**  
MS-SSTP、RFC 1661、RFC 1332、RFC 2759、RFC 3079からパケット構造定義、状態表、タイマー、試験値を作り、WindowsとSoftEtherの正解pcap記録を収集する。

**理由**  
コードより先に外部挙動と受け入れ条件を固定し、仕様適合性と実装の完了条件を客観的に検証するため。

**不採用理由: 既存Rust試作を分岐して動くところから始める**  
既存試作にはCrypto Binding欠落、証明書検証無効化、PAP中心、過大キュー等があり、基礎設計から修正する必要がある。

### 段階 1: `sstp-core` 符号化・復号処理と状態機械

**決定**  
SSTP制御・データの符号化・復号処理、ストリーム復号器、LCP、PAP、IPCP、タイマー、切断を純粋な中核として実装し、ファズ試験を同時に追加する。

**理由**  
オペレーティングシステムやTLSを接続する前に、状態機械単体の正しさを確立できる。

**不採用理由: 最初にFlutter接続画面を完成させる**  
利用者画面はプロトコルの正しさを前進させず、実装基盤の接続仕様が変わるたびに手戻りが生じる。利用者画面の模擬実装は並行可能だが本結合は後にする。

### 段階 2: Linuxの画面なし端点間試験

**決定**  
rustls、TCPの443番ポート、HTTP SSTP交渉、Linux TUN、netlinkを接続し、PAPでSoftEther、次にWindows RRASへ接続する。

**理由**  
最小構成で実パケットを流し、pcap形式の記録とログを使って層ごとに検証できる。

**不採用理由: 同時に4実行環境の接続部を作る**  
プロトコル中核が未安定な時期に同じ不具合を4環境で追うことになる。

### 段階 3: MS-CHAPv2とCrypto Binding

**決定**  
RFC 試験値、MPPE鍵導出、HLAK/CMK、SHA-256 Compound MACを実装し、Windows RRASを必須試験にする。

**理由**  
実用認証とSSTP安全性方式を完成させ、最小実用製品の中核互換性を確立する。

**不採用理由: Crypto Bindingを後期の安全性強化へ回す**  
認証接続仕様と鍵寿命へ深く関わる必須機能であり、後付けすると中心設計を変更する。

### 段階 4: データパス最適化

**決定**  
先頭余白付きバッファープール、一括処理、上限付きバイト待ち行列、書き込み処理優先度、TLS送信確定方針を実装し、`iperf3`、中央処理装置使用量、メモリ確保、常駐集合容量を測る。

**理由**  
正しい基準実装を先に作ることで、各最適化の効果と回帰を測定できる。

**不採用理由: 最初から`unsafe`による無複製へ最適化する**  
正しさの基準がなく、性能改善とプロトコル不具合を区別できない。

### 段階 5: Android、macOS、iOSの順に展開する

**決定**  
Android VpnService、macOS Packet Tunnel、iOS Packet Tunnelの順で追加する。

**理由**  
Androidはネイティブファイル記述子でLinux実装を活用しやすい。Appleは別プロセス、署名、権限設定、コールバック型パケットI/Oの制約が強いため、安定した中核をまずmacOSへ持ち込み、その後に制約の強いiOSへ展開する方が効率的である。

**不採用理由: Appleを最初の製品プラットフォームにする**  
SSTP Connectとの比較には有用だが、プロトコル不具合とNetwork Extension固有問題が混ざり、初期開発速度を落とす。

### 段階 6: Flutter利用者画面と製品機能

**決定**  
接続設定、安全な保存機構連携、状態、統計、診断情報出力、分割トンネルと全通信トンネル、自動接続をFlutter利用者画面へ実装する。

**理由**  
安定した版管理済みの制御接続仕様に対して利用者画面を作ることで、オペレーティングシステムごとの実装基盤差を吸収できる。

**不採用理由: 利用者画面から実行環境内部設定を直接操作する**  
設定の検証と適用責任が分散し、利用者画面とサービス・拡張機能の状態が不一致になる。

### 段階 7: IPv6とEAP

**決定**  
IPv6CP、二重プロトコル層、EAP-MSCHAPv2、EAP-TLS、需要に応じてPEAP、TTLS、TEAPを順次追加する。

**理由**  
基礎状態機械と鍵受け渡しインターフェースが実証された後なら、追加認証方式を局所化できる。

**不採用理由: 永久にMS-CHAPv2だけを対象とする**  
MS-CHAPv2は古い暗号基本機能を必要とし、企業環境では証明書ベースEAP-TLSの需要がある。長期的な安全性と互換性のため拡張余地を残す。

## 15. 最小実用製品完了条件

この節の条件は `TODO.md` のH8で同一候補に対して確認する。個別機能の実装完了だけでは、この節の完了を意味しない。

### 15.1 機能完了条件

**決定**  
Linux上でWindows RRASとSoftEtherへMS-CHAPv2で接続し、IPv4、DNS、全通信経路と分割経路、Echo、正常切断、再接続が動作することを最小実用製品条件とする。

**理由**  
単なる接続手順の成功ではなく、実用トンネルとして必要な一連の動作を検証できる。

**不採用理由: IPパケットが1個通った時点で最小実用製品完了とする**  
DNS、経路、長時間Echo、切断、再接続の不具合が残り、クライアントとして評価できない。

### 15.2 セキュリティ完了条件

**決定**  
TLSホスト名検証、SHA-256 Crypto Binding、秘密情報zeroize、全符号化・復号処理 ファズ試験、依存監査を必須とする。

**理由**  
セキュリティ機能を「後で追加する品質」にせず、接続機能と同じ完成条件に含める。

**不採用理由: 最初のリリースだけ証明書警告・Crypto Binding省略を許す**  
互換性のために不安全な挙動が固定化され、後の厳格化が破壊的変更になる。

### 15.3 性能完了条件

**決定**  
同一Linux環境の`sstp-client`に対して、損失率 0%で同等以上の実効処理量を持ち、中央処理装置使用量と常駐集合容量に重大な退行がなく、アイドル時ポーリング 起床を発生させないことを最低条件とする。

**理由**  
Linuxの`sstp-client`は既に実用的に速いため、現実的で厳しい基準になる。Rustという言語だけで高速性を仮定しない。

**不採用理由: 任意の絶対メガビット毎秒だけを目標にする**  
中央処理装置、ネットワーク接続部、サーバ、往復時間、TLS暗号方式で結果が変わる。比較対象を同条件に置く方が設計改善を評価しやすい。

## 16. 実装開始時の禁止事項

### 16.1 禁止リストを設ける

**決定**  
初期レビューで以下を原則禁止とする。

- `pppd`、PTY、TTYへの依存
- パケットごとの新規`Vec`生成
- パケットごとのDart/Swift/Kotlin往復
- 無制限または数万要素のチャネル
- 通常ループの固定待機/短周期ポーリング
- 複数処理単位によるTLS 書き込み処理の相互排他共有
- 1 read = 1 SSTP パケットという仮定
- 利用者画面プロセスによるVPN インターフェース所有
- 証明書検証を完全無効化する通常設定
- 中核内のunsafe
- パスワード、鍵、パケット内容の通常ログ出力

**理由**  
既存実装で確認された性能・安全・移植性の問題を、コードレビュー時に早期検出できる。

**不採用理由: ガイドラインに留め、例外を自由にする**  
短期的な動作確認のために導入された暫定処理が、そのまま製品データパスへ残りやすい。例外が必要な場合は、計測結果と安全条件を設計記録へ残して承認する。
